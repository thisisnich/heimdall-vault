## 2.1 Purpose of Project Management

### Sub-topics
- 2.1.1 Project vs Process
- 2.1.2 Time, Cost and Scope Interactions

### Overview

A **project is defined as a unique effort with a defined beginning, defined end, specific deliverables and defined resources**. The full cycle of a project involves the following phases:

- Project initiation and organising
- Project planning
- Project execution
- Project control and monitoring
- Project closure

Project management enables the project manager to complete the project **on time**, **finish within budget** and making sure that **customers' satisfactions are achieved**.

Project management **tools** can be used to ensure that the proper knowledge and resources are available when and where needed and above all to ensure that the expected outcome is produced in a timely and cost effective manner.

---

## 2.1.1 Project vs Process

Unlike building a standard house (a business process), a project is analogous to building a **custom house**.

| Project                                    | Business Process                                      |
| ------------------------------------------ | ----------------------------------------------------- |
| Has a beginning and an end                 | The same process is repeated over and over again      |
| Produces a different output or deliverable | Produces the same output each time the process is run |
| Has no regular work assignments            | Has regular work assignment                           |

---

## 2.1.2 Time, Cost and Scope Interaction

An equilibrium has to be maintained among the following **3 constraints** when managing changes in a project:

- **Time** (e.g. develop a new inventory tracking system by the start of next year)
- **Cost** (e.g. you will receive $80,000.00 for the inventory tracking system)
- **Scope** (e.g. the new system should be developed using MS .NET technology and integrated with an existing SAP system for enterprise resource planning)

The quality of the deliverables may be affected if these constraints are not handled properly.

### Quiz

> *1. What happens when the duration of the project is decreased?*
> **Solution:** Increase the cost and/or decrease the scope.

> *2. What happens if the budget (cost) of the project is decreased?*
> **Solution:** Increase the project duration (reduce the manpower) and/or decrease the scope.

> *3. What happens if project scope is increased?*
> **Solution:** Increase project cost and/or increase project duration.

## 2.2.1 Project Initiation & Organisation

### Organise a Project Team

**Benefits of initiating and organising:**
- A solid organisation in place
- An understanding of top-down goals
- A frame for team interaction and project processes

**Project Initiation involves getting the project authorised.** A project is authorised as a result of one of the following:

- **Technology advances** (e.g. a car company authorises a project to create an efficient car in response to gasoline shortages)
- **Social needs** (e.g. a utility company authorises a project to build a new substation for serving a new industrial park)
- **Market demands** (e.g. an electronic firm authorises a project to develop a video game player)
- **Business needs** (e.g. a paint manufacturer authorises a project to establish guidelines for handling toxic materials)
- **Customer's requests** (e.g. a NGO authorises a project to provide portable water and sanitation education to cholera-affected communities in remote areas)

### Identifying and Selecting Projects

**Feasibility** is a term used to describe how achievable various project options are. Feasibility studies decide which proposal to choose from a list of options. Types of feasibility studies include:

- **Technology feasibility** — checklist includes: development (technology thoroughly tested), applicability (technology must satisfy objective), ease of use, degree of disruption during construction/installation, feature analysis
- **Social feasibility** — social impacts on affected groups must be evaluated; issues like redundancy caused by computerisation must be addressed
- **Ecological feasibility** — customers may prefer environmentally sound alternatives
- **Business goal fit** — high financial gain proposals may divert company from primary goals
- **Financial feasibility** — considerations include: whether capital or revenue, value of project, when it occurs, whether one-off or recurring

### SWOT Analysis

To identify a suitable project, the **SWOT** analytical tool can be used (Strengths, Weaknesses, Opportunities, Threats):

- **Strengths** — competitive advantage, skills and competency needed, effect on staff morale
- **Weaknesses** — skills lacking, IT system not producing expected information
- **Opportunities** — external events/changes that can be exploited to the organisation's advantage
- **Threats** — external events/changes that must be defended against (e.g. loss of market share to competitors)

### Form a Project Team

The 3 steps for organising a project team:
1. Identify a **sponsor**
2. Appoint a **project manager**
3. Identify **core team**

#### Identify the Sponsor

The sponsor is the hardest role to find (e.g. managing director). Responsibilities include:
- Appoint a project manager (PM)
- Set strategic direction
- Approve the plan and budget
- Support team members
- Approve project changes and review progress
- Set priorities and resolve conflicts

#### Appoint a Project Manager (PM)

The PM establishes and leads a team to achieve project objectives within the boundaries of schedule, resources, and scope. PM responsibilities include:

- Define project management process
- Prepare and obtain management approval of project plan
- Ensure all team members understand and accept their responsibilities
- Expend project resources according to approved project plan
- Analyse actual performance against plan
- Establish priorities among project activities
- Provide periodic status reports
- Coordinate management and technical decisions
- Resolve conflicts and interface problems within project

#### Identify Core Team

- Get commitment and buy-in from manager(s) of team members
- Ensure they have time required for the project
- Get members' commitment to the project
- Ensure members can make decisions and commitments for their organisation
- Core team should have **at most 7 members**
- Core members make major decisions and recommendations
- Extended members provide specific expertise (no limit to number)
- Missing critical skills will cause serious problems during the project

### Project Organisation Structures

Two common structures:

1. **Functional Organisation** — each project has a PM who draws resources from functional departments (HR, Engineering, QC, Production)
2. **Matrix Organisation** — projects (rows) intersect with functional departments (columns), creating a grid structure

## 2.2.1 (continued) — Define Project Parameters

The purpose of defining project parameters is to **ensure common understanding of the project and objectives among all team members**. Key documents include:

### Project Objective Statement

States **what** you are going to do (scope), **by when** (schedule), and **with how many** (resources/costs). Should be at most 25 words with only 1 target, written in clear and precise language.

> *Example: "Implement an integrated employee system database by 31 Dec 2006 with a budget of S$20 million."*

### Flexibility Matrix

Helps the team determine trade-offs among scope, schedule and resources:

| | Least Flexible | Moderately Flexible | Most Flexible |
|---|---|---|---|
| Schedule | | ✓ | |
| Scope | ✓ | | |
| Resource | | | ✓ |

- **Least flexible** — cannot be exceeded
- **Moderately flexible** — can be maximised or minimised within project objective
- **Most flexible** — management willing to exceed original goal if necessary

### List of Major Deliverables (MD)

Defines what the project **is expected to produce**. Provides scope and boundaries for the project team and reduces rework cycle.

*Example — Setting up a New Restaurant:*

| MD1 (Furniture) | MD2 (Equipment) | MD3 (Dining Area) |
|---|---|---|
| Furniture sourced | Equipment sourced | Layout completed |
| Furniture purchased | Equipment purchased | Contractor sourced |
| Furniture delivered | Equipment delivered | Work completed |
| | Equipment installed | Furniture installed |

### Project Charter

A **document that formally authorizes a project**. Provides the PM with authority to apply organisational resources to project activities. Contains:

- Project title and description
- Project manager assigned and authority level
- Project objective statement
- Business case (why is the project done?)
- Product description/deliverables

**Validation & Concurrence required from:**
- **Team** — approve project definition, document all assumptions, provide clear understanding of project description and priorities
- **Management** — present project description, objective statement, major deliverables, and flexible matrix for approval to proceed

## 2.2.1 (continued) — Establish a Project Framework

A **framework defines agreements made by the project team as to how the team will operate**. It clarifies expectations and ensures important decisions are made early and documented.

A project framework should encompass **4 areas**: Planning, Tracking, Practices, Relationships.

The framework should address **4 questions**: What is it? Who does it? Why do it? How is it done?

### Planning Decisions Checklist

*What does the plan include?*
- Does it meet customer's/management's needs?
- What product or service life cycle does it follow?
- What are the outputs from the planning process?
- What project management tools will be used? (e.g. PERT, CPM, Gantt Chart)

*Who is responsible for developing the project plan?*
- If more than one person, who should be involved?
- What are the specific roles and responsibilities of each participant?

*Who needs to be informed of decisions made?*
- How will they be kept informed?
- Will a list of terms be distributed for better comprehension?

*Is there a plan to continuously improve management of the project?*
- Is there past experience on similar projects?
- How can we use them?

### Tracking Decisions Checklist

*What types of meetings will we hold?* (general, top level, activity specific, etc.)
- Where and how often will they be held?
- Who will run them and produce agendas?

*How will we get data from team members about progress?*

*How will we assess progress?*
- Through **Progress Review Meetings**

*What reports will be generated?*
- Who will generate, analyse and distribute them?
- To whom should they be distributed?

### Practices Decisions Checklist

*What should the project file contain?*
- How will project information be kept? Who will create/maintain it?

*When will the plan baseline be frozen?*
- Who decides? What will the decision be based on?

*Who will have change approval authority?*
- What documents for change approval? How are changes documented?

*Will a project assessment (continuous improvement) meeting be held?*
- Who attends, how often, what subjects?

*What is the release plan process?*
- Who owns it, by when should it be defined, who approves it?

### Relationship Decisions Checklist

*What departments or organisations will we need to interact with?*

*What are the roles and responsibilities of each dept?* (reviewer, approver, creator, etc.)

*How do we communicate?*
- Among ourselves and with others outside the team?
- How frequently?

*What is the process to resolve disagreements?*
- Conflict resolution process? Standard decision making process?

*What is the escalation process for making decisions?*
- Who has final decision making authority?
- Who decides when to escalate a problem?

## 2.2.2 Project Planning

### Develop Work Breakdown Structure (WBS)

WBS is a tool for identifying and presenting project work in a **hierarchical and logically structured manner**.

**Purpose:** Ensure that all work required to meet the objectives is properly identified, defined and assigned.

**Key elements:**
- Sequence is **not** important
- Completeness **is** important
- Verb/noun symbolises task & activity
- Each breakdown needs a minimum of 2 tasks

**WBS Development Process:**
- Use a **top-down approach** to identify 4 to 7 major components
- Group by: product deliverables, life cycle phases, functional responsibilities, or geographical location
- Validate using a **bottom-up approach**
- WBS development is a **team process** — not done by the PM alone

**WBS Format and Coding:**
- Coding allows for easy reference and reduces confusion during discussions
- Typical coding structure is numeric (e.g. 1.0, 1.1, 2.1.1)
- Formats: Graphical (tree diagram) or Outline (indented list)

### Approximate Duration of Tasks

Duration approximation relies on **previous experience and documented history**. Key principles:

- Durations are assigned by **owners** of the tasks
- Padding should be avoided — use the **most likely duration**
- All assumptions should be documented
- Number of persons and training requirements should be documented

**Relationship:** Effort → Duration → Calendar Time
- **Effort** — for charging purposes
- **Duration** — for schedule purposes (consider productivity & wait time)
- **Calendar Time** — real life planning for tracking purposes (consider non-working days, weekends, holidays)

### Generate Gantt Chart

A Gantt Chart is a **simple representation of a project** showing the time taken and resources required for each activity at any point in time.

It is a useful **communication tool** which is easy to understand and provides an overview of responsibility and progress on the project.

## PERT and CPM

### Overview

**PERT (Program Evaluation and Review Technique)** and **CPM (Critical Path Method)** are two project management techniques used for **scheduling, planning, and controlling** complex projects. Both methods help project managers identify the **critical path** — the sequence of tasks that determines the minimum project duration.

### PERT (Program Evaluation and Review Technique)

**Purpose:** Used when task durations are **uncertain** or variable. PERT uses **probabilistic time estimates** to calculate expected project duration.

**Key Features:**
- Uses **three time estimates** for each activity:
  - **Optimistic (O)** — Best-case scenario (everything goes perfectly)
  - **Most Likely (M)** — Normal scenario (most probable duration)
  - **Pessimistic (P)** — Worst-case scenario (everything goes wrong)
- Calculates **expected time (TE)** using the formula:
  - **TE = (O + 4M + P) / 6**
- Provides a **probability distribution** for project completion
- Best for **R&D, new product development, IoT prototyping** where uncertainties are high

**PERT Formula for Variance:**
- **Variance (σ²) = [(P - O) / 6]²**
- **Standard Deviation (σ) = (P - O) / 6**

**Example — IoT Sensor Development:**
- Optimistic: 3 days
- Most Likely: 5 days
- Pessimistic: 13 days
- **TE = (3 + 4×5 + 13) / 6 = (3 + 20 + 13) / 6 = 36 / 6 = 6 days**
- **σ = (13 - 3) / 6 = 10 / 6 ≈ 1.67 days**
- **σ² ≈ 2.78**

### CPM (Critical Path Method)

**Purpose:** Used when task durations are **deterministic** and well-known. CPM focuses on identifying the **critical path** — the longest path through the network diagram that determines the project duration.

**Key Features:**
- Uses **single time estimate** for each activity (deterministic)
- Identifies **critical activities** — activities with zero slack/float
- Calculates **earliest start (ES)**, **earliest finish (EF)**, **latest start (LS)**, and **latest finish (LF)** times
- Determines **slack/float** — amount of time an activity can be delayed without delaying the project
- Best for **construction, manufacturing, well-defined projects**

**CPM Calculations:**
- **Forward Pass:** Calculate ES and EF (left to right)
  - ES = max(EF of predecessors)
  - EF = ES + Duration
- **Backward Pass:** Calculate LS and LF (right to left)
  - LF = min(LS of successors)
  - LS = LF - Duration
- **Slack/Float = LS - ES** or **LF - EF**
- **Critical Path:** Activities with **zero slack**

### Comparison: PERT vs CPM

| Aspect | PERT | CPM |
|---|---|---|
| **Time Estimates** | 3 estimates (O, M, P) | 1 estimate (deterministic) |
| **Uncertainty** | High uncertainty | Low uncertainty |
| **Focus** | Probability of completion | Critical path identification |
| **Best For** | R&D, new projects, IoT prototyping | Construction, manufacturing |
| **Complexity** | More complex calculations | Simpler calculations |
| **Output** | Expected time + variance | Critical path + slack |

### Network Diagram (PERT/CPM)

A **network diagram** (also called **activity-on-node** diagram) visually represents:
- **Activities** as nodes/boxes
- **Dependencies** as arrows
- **Sequence** of tasks

**Types of Dependencies:**
- **Finish-to-Start (FS)** — Task A must finish before Task B starts (most common)
- **Start-to-Start (SS)** — Task A must start before Task B can start
- **Finish-to-Finish (FF)** — Task A must finish before Task B can finish
- **Start-to-Finish (SF)** — Task A must start before Task B can finish (rare)

### Critical Path in IoT Projects

For an IoT project, the critical path might include:
1. **Hardware selection** → 2. **Sensor procurement** → 3. **Circuit design** → 4. **PCB fabrication** → 5. **Firmware development** → 6. **Integration testing**

If any activity on this path is delayed, the **entire project is delayed**.

**Example IoT Project Critical Path:**
```
Task A: Hardware Selection (2 days)
Task B: Sensor Procurement (5 days) — depends on A
Task C: Circuit Design (4 days) — depends on B
Task D: PCB Fabrication (7 days) — depends on C
Task E: Firmware Development (8 days) — depends on D
Task F: Integration Testing (3 days) — depends on E
Total Critical Path: 2 + 5 + 4 + 7 + 8 + 3 = 29 days
```

### Slack/Float in IoT Projects

Activities **not** on the critical path have **slack** — they can be delayed without affecting project completion.

**Example:**
- **Documentation** can be done in parallel with development
- **User interface design** may have slack if backend development is critical
- **Packaging design** often has slack as it's done near the end

### Using PERT/CPM for IoT Projects

**When to use PERT for IoT:**
- Developing **new sensor technology** with unknown performance
- **Wireless protocol selection** with uncertain range/reliability
- **Cloud integration** with third-party services
- **Regulatory certification** with unpredictable timelines

**When to use CPM for IoT:**
- **Hardware assembly** with known component lead times
- **Software testing** with established test suites
- **Documentation** with clear requirements
- **Packaging and shipping** with standard processes

**Hybrid Approach:**
Many IoT projects use **PERT for R&D phases** and **CPM for production phases**:
- **Design/Prototype phase:** PERT (high uncertainty)
- **Development phase:** PERT for new tech, CPM for standard tasks
- **Testing phase:** CPM (well-defined test procedures)
- **Production/Deployment:** CPM (standardized processes)

## 2.2.3 Project Execution, Monitoring & Control

Project execution, control & monitoring consists of:
- **Monitoring** the environment (collect info & update status)
- **Manage** change (analyse variance)
- **Track** progress (take adaptive action when necessary)
- **Communicate** progress (report project status)

**Benefits:**
- Provides a clear indication of progress
- Keeps everybody informed
- Encourages addressing all problem areas
- Provides credible completion estimates

### Collect Data & Update Status

Collecting info and updating status ensures:
- Problems are tracked early
- Progress is visibly demonstrated
- Communication is ongoing

### Plan and Take Adaptive Action

When actual activities **deviate from planned activities**, this is known as **variance**. After identifying causes and impacts, the PM must plan for adaptive action.

**When planning for adaptive action:**
- Involve the team members in the planning process
- Review project change control process, framework, risk management plan
- Re-visit all assumptions (schedule, resource, scope)
- Check status data for accuracy
- Implement contingency plan
- Hold key milestones while re-arranging task schedule
- Re-plan/re-estimate schedule or resource plan

**When taking adaptive action:**
- Implement new plan
- Communicate any changes to team members
- Get buy-in from project team
- Document all changes

**Follow up:** Changes in one aspect (scope, schedule, resource) **will affect others** — monitor carefully.

### Report Project Status

Objective: Provide information about the status of the project, focusing on **essential information for successful project decision making**.

When communicating project status:
- Do it **at least once a month** — management and customers do not like surprises
- State **overall % of completion** — emphasise milestones
- State **major tasks completed** during this period
- Provide **project analysis and highlight any variance and adaptive action** taken
- State **major tasks for next period** (next 2 weeks to next month)
- Include outstanding issues: updated schedules, updated milestones, updated cost spreadsheet

## 2.2.4 Project Closure

After the customer accepts the final deliverables, the PM must still go through the **process of project closure**.

Key considerations:
- **Announce** end of project
- **Conduct** final project review
- **Debrief** key learning
- **Deliver** acknowledgements and rewards
- **Closing ritual** (celebration)
- **Complete paperwork:**
  - Contract
  - Payment
  - Schedule update
  - Document lessons learnt
  - Close project file

### Conduct Review

Conduct a review that highlights both **what went right and what went wrong**. Purpose is **not finger pointing** but to minimise problems in future.

- Essential for **continuous improvements**
- Identify elements contributing to project success
- Avoid problems in future and identify processes needing improvement
- **Capture key learning** which are invaluable for future project planning

### Close the Project File

- Complete all final review documents
- Bundle all status reports and label them to ensure individual papers are not lost
- Send all project files for archiving
- PM is responsible for ensuring all project files are stored properly

## 2.3 Aerospace System Development Process

### Major Milestones in the Avionics Development Process

The majority of aircraft systems have electronics (**hardware**) associated with the control function, usually controlled or driven by **software**. The development process typically follows **5 major milestones**:

1. **System Requirement Review (SRR)**
2. **System Design Review (SDR)**
3. **Software Specification Review (SSR)**
4. **Preliminary Design Review (PDR)**
5. **Critical Design Review (CDR)**

```
Perceived        SRR          SDR          PDR         CDR       Hardware
System Reqt  ─────────> PSD ──────> Reqt ──────> Prelim ──────> Detailed ──────> Build ─┐
                                   Analysis     HW Design       HW Design               │
                                      │                                                  ▼
                                      └──────> Reqt ──────> Prelim ──────> Detailed ──> Code ─> Integration
                                              Analysis      SW Design       SW Design    & Test   & Test ──> Final
                              SSR ─────────────────^            PDR (SW)    CDR (SW)              Product
```

### System Requirements Review (SRR)

- A **top-level multidisciplinary review** of the perceived system requirements
- Successful completion leads to a **Preliminary System Design (PSD)**
- Typical considerations include:
  - Functions
  - Software code size
  - Operating hours, MTBF
  - Dimensions, size, weight
  - Power requirements (e.g. 28VDC, 115VAC, 400Hz 3-phase)
  - Environmental factors (temperature, pressure, humidity, etc.)

### System Design Review (SDR)

- Follows immediately after the PSD phase
- A **top-level review of system hardware characteristics** so that preliminary design may proceed with confidence
- Key hardware characteristics are reviewed to ensure **no major mismatches** between system requirements and hardware capabilities

### Software Specification Review (SSR)

- Similar process to hardware SDR, but focuses on **software requirements analysis**
- Identifies any limitations such as throughput, timing or memory imposed by the hardware solution
- Both SDR and SSR allow the preliminary design to develop up to the **Preliminary Design Review (PDR)**

### Preliminary Design Review (PDR)

- **First detailed review** of the initial design (both hardware and software) versus system requirements
- Includes **preliminary tests and/or software simulation**
- Usually the **last review before committing major design resources** to the detailed design process and committing major program resources and investment

### Critical Design Review (CDR)

- By CDR, **major effort, resources, and investment** will have been committed to the program
- Identify **final design flaws**, or trade-off risks of one path implementation versus another
- **Last opportunity to review and alter the direction of design** before very large commitments and final design decisions are made
- Major changes in system design **after CDR** will be **very costly** in terms of cost and schedule

### Integration and Test Phase

- Both hardware build and software coding are **integrated and tested** leading to **final product realization**
- During the test phase (also known as **qualification**), the final product is subject to a rigorous test program to verify its fitness for purpose — includes analysis, modelling, functional testing on ground or flight trials
- Even after system validation and certification or when product is in service, there could be **product modifications** for performance, reliability, or maintainability reasons
